# Архитектура Web FTP Client

## Обзор
Приложение представляет собой FTP-клиент с веб-интерфейсом. Оно позволяет просматривать локальные файлы на сервере, где запущен клиент, а также подключаться к удалённым FTP-серверам и управлять их файлами. 

Проект построен по классической архитектуре клиент-сервер, где большую часть работы выполняет серверная часть, а клиентская отображает данные с использованием современных подходов (HTMX, SSE).

## Стек технологий
### Backend
- **Язык**: Rust
- **Веб-фреймворк**: Axum (асинхронный веб-фреймворк поверх Tokio).
- **FTP-клиент**: `suppaftp` (асинхронная библиотека для работы с протоколом FTP).
- **Шаблонизатор**: `askama` (типизированный шаблонизатор, компилирующий HTML в Rust-код на этапе сборки).
- **Асинхронный рантайм**: `tokio`.

### Frontend
- **HTML/CSS**: Обычные формы и стили (ванильный CSS из `assets/css`).
- **Интерактивность**: `HTMX` используется для AJAX-запросов (например, `hx-get`, `hx-post`) без написания пользовательского JavaScript, обновляя лишь фрагменты страницы.
- **События реального времени (SSE)**: Используется расширение HTMX SSE (`sse.js`) для получения статуса подключения с сервера (обновление подвала и кнопки подключения).

## Структура приложения

### 1. Состояние приложения (`AppState`)
Определено в `main.rs`. Хранит глобальное состояние, разделённое между всеми обработчиками запросов:
- `connection`: `Arc<Mutex<Option<AsyncFtpStream>>>` — текущее FTP-соединение (если установлено).
- `local_path`: `Arc<Mutex<PathBuf>>` — текущая директория локальной файловой системы, которую просматривает пользователь.

### 2. Маршрутизация и обработчики (`routes.rs`)
В приложении описаны следующие основные эндпоинты:
- **`GET /`** (`index`): Отдает главную страницу-оболочку (`index.html` -> `base.html`).
- **`GET /local_list`**: Читает содержимое `local_path` и возвращает скомпилированный HTML-фрагмент с таблицей файлов (`local_files_table.html`).
- **`POST /local_change_directory`**: Изменяет `local_path` (входит в папку или выходит наверх `..`) и запрашивает перерисовку списка локальных файлов.
- **`POST /connect`**: Принимает данные формы (хост, порт, логин, пароль), устанавливает FTP-соединение при помощи `suppaftp` и сохраняет его в `AppState`.
- **`POST /disconnect`**: Разрывает FTP-соединение и очищает его из состояния.
- **`GET /list`**: Получает список файлов из текущей удаленной директории по FTP и возвращает HTML-таблицу (`files_table.html`).
- **`POST /change_directory`**: Отправляет FTP-команду `cwd` (переход в папку) или `cdup` (переход на уровень выше), обновляет состояние и запрашивает перерисовку удаленных файлов.
- **`GET /events`**: Эндпоинт Server-Sent Events (SSE). Каждые 2 секунды пушит на клиент HTML-обновления о том, подключен ли FTP-сервер (обновляет footer и кнопку "Подключиться/Отключиться").

### 3. Шаблоны (`templates/`)
- `base.html`: Основной каркас с подключением HTMX, CSS и определением формы подключения.
- `index.html`: Наследует `base.html`, определяет две главные панели: "Локальный сервер" и "Удалённый сервер".
- `local_files_table.html` / `files_table.html`: Шаблоны для генерации таблиц со списком файлов. Обеспечивают навигацию через атрибуты HTMX (click по директории вызывает `hx-post="/change_directory"`).

## Поток данных (Data Flow)
1. Пользователь заходит на главную страницу (загружается `base.html` + `index.html`).
2. HTMX делает автоматические запросы `hx-trigger="load"` к `/local_list` и `/list`, подгружая списки файлов.
3. Клиент устанавливает SSE-соединение с `/events`, чтобы реагировать на изменения статуса подключения.
4. При клике на локальную папку, отправляется POST-запрос, сервер меняет глобальный путь и возвращает HTMX-триггер для обновления локального списка файлов.
5. При заполнении формы FTP и отправке, сервер пытается подключиться. В случае успеха, статус обновляется, и отображаются удаленные файлы.

## Дальнейшее развитие
В текущей реализации хранение состояния подключения FTP в `Arc<Mutex<>>` означает, что состояние является глобальным для всего сервера. Если клиентом будут пользоваться несколько пользователей с разных браузеров, они будут делить одно FTP соединение и одну локальную папку. Для поддержки многопользовательского режима, `AppState` должен быть привязан к сессиям (например, cookie на базе UUID).
